# AxePilot — Project Rules & Guidelines

These rules apply to all development on this project, regardless of who (or what) writes the code.

## 1. Development Process & Testing

- **Test-Driven Development (TDD)**: Every new feature, component, or safety/security-relevant change starts with failing tests. Write the tests first (confirm they're actually red — compile error or failing assertion, not just "haven't run yet"), then the implementation that makes them pass.
- **Testing tool**: Unity, via `./.venv/Scripts/pio.exe test -e native` (native, host-side — no ESP32 hardware required). This is the only environment tests run in.
- **Test isolation**: Tests must be self-contained and must not depend on — or mutate — live infrastructure (the physical CYD, a real Bitaxe, real Telegram/DeepSeek endpoints, real WiFi). Hardware is mocked through the `interfaces/` layer (`IDisplay`, `IHttpClient`, `ISystemTime`, `ISystemInfo`, `ISettingsStore` → `test/mocks/Mock*`). The suite must pass on a clean checkout with no device connected; "I flashed it and it looked fine" is not a substitute for a test.
- **Static quality gates**: All code must pass `pio check -e esp32-cyd --skip-packages --fail-on-defect=medium --fail-on-defect=high` (cppcheck) with zero medium/high findings in `src/`/`include/`. There is no C++ equivalent of a mypy/tsc-style type checker in this project's toolchain — the compiler itself (a successful `pio run -e esp32-cyd` build) is the type-correctness gate, and no formatter (e.g. clang-format) is enforced; this was a deliberate choice, not an oversight (revisit only on explicit request). Suppress a cppcheck rule only when genuinely unavoidable (see `platformio.ini`'s `check_flags` for the two current suppressions — `unusedFunction` and vendored `libdeps` — both with a comment explaining why) and keep the suppression as narrow as possible.

## 2. GitHub Hygiene & CI/CD

- **Issue tracking**: Every new phase or major feature gets a dedicated Issue before work begins. Small fixes (CI repairs, typos, test-only patches) may go straight to a PR.
- **Milestones**: Group issues into logical Milestones (this project uses: UI & Touch · Reliability & Network · Tech Debt & Tests · AI & Insights).
- **Language & quality**: All issues, milestones, PR descriptions, and review comments are written in **English**. Chat/conversation with the project owner is in Ukrainian — these are separate channels; don't mix them. Issues must include detailed, verifiable acceptance criteria.
- **Fresh main/default branch**: Before creating a branch, or before any operation that reads "the current state of `master`" (computing a diff or merge-base, checking whether a branch is an ancestor), run `git checkout master && git pull` first — never rely on a possibly-stale local ref. (A stale ref can silently reintroduce a dropped rule during any history-rewriting or branch-cleanup operation.)
- **PRs**: Every PR carries a detailed description with these explicit sections:
  - **What** — what it does and which issue it closes, if any.
  - **Where** — a high-level map of which parts of the codebase were touched, without deep technical detail — that belongs in How.
  - **Why** — the rationale and the value the change brings.
  - **How** — technical implementation details, scope, and notable decisions.
  - **Test results** — proof of testing: local native-suite result (X/X), `pio check` result, `pio run -e esp32-cyd` build result + Flash/RAM %, and the CI pipeline status.
- **Development & closure workflow** (every change goes through all steps, in order):
  1. **TDD development** — tests first, then implementation.
  2. **Commit & push** — push the changes to a feature branch, named `<type>/<issue-number>-<slug>` (`feat`/`fix`/`docs`/`chore`/`refactor`/`test`/`perf`/`ci`).
  3. **Green build & description** — wait for a green pipeline and complete the PR description, including test results.
  4. **Self-review** — performed by the AI assistant on behalf of the project owner (solo project, no separate human reviewer): review the full diff between the merge-base and the branch tip (`git diff master...branch`), not just a recollection of what changed, then post a detailed review comment via `gh pr comment` explicitly stating a self-review was performed, covering correctness, scope, and test evidence — **on every PR, even when nothing was found** — including an explicit check of the gitleaks report via the GitHub Actions API/logs (not just the check's pass/fail status shown by `gh pr checks`, in case the workflow is ever reconfigured to not fail the pipeline on a hit). If issues are found, fix in a new commit, document it in the same PR comment thread, then re-review. *(There is deliberately no separate formal `gh pr review --approve` step: GitHub's API refuses to let an account approve its own PR — `Can not approve your own pull request` — and every PR in this repo is authored under the owner's own account/token, with no separate bot identity. The self-review comment is the actual review record; don't reintroduce an approve step without first standing up a distinct GitHub identity for it.)*
  5. **Merge & close** — once the pipeline is green and self-review has passed, merge via `gh pr merge <N> --squash --delete-branch`. The related issue is closed only after the successful merge — confirmed via `gh issue view <N> --json state` that the issue's state is actually `CLOSED`, not assumed from a `Closes #N` keyword alone (auto-close has silently failed to fire in the past for reasons that weren't investigated at the time — verify, don't assume). If GitHub already auto-closed it, still post a closing summary comment on the (already-closed) issue rather than skipping it.
  6. **Verify main** — `gh run list --branch master --limit 1` (or equivalent) to confirm the resulting pipeline on `master` itself is green before moving to the next task; a merge is not complete if the default branch goes red.
  7. **Demo** — for any merge that introduces or changes runtime behavior (skip for pure docs/process changes): before moving to the next task, present a short demo covering a plain-language summary of what was built and why (not a re-paste of the PR description), a diagram for anything with a flow/architecture/sequence worth visualizing, and — since this is embedded firmware — flash the merged build to the physical device (prefer OTA: `pio run -e esp32-cyd-ota -t upload`; serial `-e esp32-cyd -t upload --upload-port COM<N>` only when required, e.g. a partition-table change) and demonstrate the feature working end-to-end on real hardware. A green test suite alone does not satisfy this step for UI-facing or hardware-facing changes.
- **Keeping agent instructions in sync**: This file is the single source of truth for process. Any root-level agent-instruction files (`CONTINUATION_PROMPT.md` and any per-tool pointer files) carry only a short summary and must be updated in the same PR whenever this section changes — a summary that drifts from this file is a bug. **Never relocate or replace a shared instruction file to fix one tool's discovery** — if a tool needs its instructions at a specific non-standard path, add a thin duplicate there; never move or delete the canonical copy.
- **Documentation accuracy**: Any claim about current code or process state — in docs, comments, or agent-instruction files — must be verified against the actual code or version-control history before being written, not against memory, assumptions, or prior session summaries. (This project's docs went stale for a full week of active development before being caught and corrected — see #82.)
- **Commit hygiene**: Commit messages must not carry AI-attribution trailers (no `Co-Authored-By: Claude`) or any other tool-generated signatures. Conventional, descriptive messages only, using the repository's already-configured git identity (`rizon0503 <coding.enthusiast@outlook.com>`) — never invent one.
- **Branch protection**: Direct pushes to `master` are strictly forbidden (enforced: `enforce_admins: true`). All changes land through feature branches and PRs only.
- **CI configuration**: Prefer official/community-maintained Actions over hand-rolled jobs that wrap security/scanning tools manually — this project already follows this for secret scanning (`gitleaks-action`, not a hand-rolled `gitleaks detect` shell step).

## 3. Package & Dependency Management

- **Tooling**: PlatformIO (`platformio.ini`) is the single package/project manager. Libraries are pinned via `lib_deps`; the CLI itself lives in the project-local venv (`./.venv/Scripts/pio.exe` — **not** on the global PATH; a bare `pio` will fail with "command not found").
- **Structure**: Single-package layout, two PlatformIO environments in one `platformio.ini`:
  - `esp32-cyd` — the real firmware (ESP32-2432S028R / Cheap Yellow Display target).
  - `native` — host-side unit tests only (`test_build_src = yes`, `build_src_filter = +<core/>`); links only `src/core/`, never `main.cpp` or `src/hal/`.

## 4. Architectural & Infrastructure Constraints

- **Target environment**: Embedded firmware on an ESP32-2432S028R (CYD). Flashed via OTA (`pio run -e esp32-cyd-ota -t upload`, ArduinoOTA, password-gated) as the default path, or serial (`-e esp32-cyd -t upload --upload-port COM<N>`) as fallback and the only option for a partition-table change. Keep the architecture portable within the Arduino/ESP32 ecosystem; avoid CYD-board-specific hacks outside `src/hal/`.
- **Strict layering**: `src/core/` = pure, natively-testable business logic (no Arduino/ESP-IDF headers). `include/interfaces/` = hardware contracts. `src/hal/` = ESP32 implementations of those contracts. `test/mocks/` = test doubles. Dependencies flow through interfaces only — `core/` must never `#include` anything from `hal/`.
- **`main.cpp` is orchestration only**: `loop()` (core 1) dispatches touch and calls into `core/UiRenderer` for all drawing; `networkTask()` (core 0, FreeRTOS, pinned) owns all HTTP(S) I/O, Telegram polling, the AI autopilot tick, OTA (`ArduinoOTA.handle()`), and the Web UI server. Never draw or make a network call directly from `main.cpp`.
- **Cross-core sharing** only via `std::atomic` or the spinlock-snapshot pattern (`MainScreenSnapshot`/`dataMux`). Data that must be read together must be copied inside ONE critical section — two separate sections produce a read-tear between network-task ticks (this happened once with sparkline data and telemetry).
- **One data path per datum**: new consumers (Web UI, Telegram, future screens) read the existing cross-core snapshot; never add a second/parallel path into `core/` state.
- **No Arduino `String` in hot loops** (`networkTask()`, `loop()`) — `snprintf` into a stack `char` buffer for formatting; `std::string` is fine for HTTP payloads and API returns. No `delay()` outside `setup()` — use `millis()`-delta state machines.
- **Rendering is dirty-checked**: `UiRenderer` repaints a line/graph only when its content actually changed since the last call, to avoid full-screen flicker. Large static assets (the Web UI's dashboard HTML) live in PROGMEM, not RAM.

## 5. Security Context

- **Secrets**: Never hardcode secrets/keys/tokens. Real credentials live only in `include/secrets.h`, which is gitignored; `include/secrets.h.example` is the tracked placeholder template. CI and every published release binary build exclusively from the placeholder — a release `.bin` is a rollback reference point, never a flashable-as-is artifact, specifically so a public release can never leak real credentials. Secret-detection scanning (gitleaks) runs on every PR and is a required status check.
- **Settings-change gate**: Every code path that changes miner frequency/voltage — AI autopilot, AI chat, Telegram `/set`, `/fan`, on-screen presets, benchmark — MUST validate through `Limits::isValidSetting()` before being sent to the miner. This is the single point that keeps a hostile or hallucinated AI response, or a prompt-injected chat message, from doing more than picking a bad-but-safe value. No new settings-writing path may bypass it.
- **TLS**: Root CAs are pinned for known cloud endpoints (`RootCerts.h` — Telegram, DeepSeek) rather than `setInsecure()`. A user-configured custom `AI_BASE_URL` can pin its own CA via `AI_ROOT_CA` in secrets.h; if it doesn't, the insecure fallback is visible (a `log_w` fires once per boot), never silent.
- **Network-facing surfaces are read-only or authenticated by design**: the Web UI (`GET /`, `GET /api/status`) is intentionally read-only with no auth, scoped to the LAN — do not add a state-changing route without an authentication story first. OTA requires a password (`OTA_PASSWORD`) matched between the running firmware and the uploader. Both are documented accepted-risk trade-offs, not oversights — see `SECURITY.md` before changing either.
- **Response size gating**: every HTTP response body (Telegram, DeepSeek, AxeOS) is rejected via `Limits::MAX_JSON_RESPONSE_BYTES` before it reaches `deserializeJson()`, bounding attacker-controlled parser allocations.
- **`SECURITY.md` is the living threat model** — LAN-attacker model, hardened-by-design properties, and the accepted-risk table (unauthenticated LAN telemetry, ArduinoOTA's MD5 auth, no flash encryption, plain-HTTP AxeOS API by upstream design, bot-token trust). Read it before touching any network-facing surface; a change that looks like a security improvement might invalidate a documented trade-off, and a change that looks harmless might silently weaken a "hardened by design" property.

## 6. Escalation — When to Stop and Ask

Some actions are high-blast-radius or hard to reverse. Proceed only with the owner's explicit, current confirmation for these, even when the general workflow above is otherwise fully authorized:

- Changes to the partition table (`partitions_two_ota.csv`) or other low-level flash layout — the current two-OTA-slot scheme was itself a deliberate architectural trade-off; changing it again needs the same kind of discussion, plus a mandatory one-time serial reflash.
- Credential or secret rotation (Telegram bot token, DeepSeek API key, OTA password) — that's a manual action in an external dashboard plus a `secrets.h` update, not something to automate.
- Any git/version-control history rewrite (rebase of shared history, force-push, history-filtering tools, repository cleanup) — this is a public repository; verify what's actually at risk (grep the full history for real secret patterns) before assuming a rewrite is warranted, and confirm with the owner regardless.
- GitHub security-setting changes (branch protection, merge/approval rules, Actions secrets).
- Touching a surface `SECURITY.md` lists as an accepted risk (e.g. adding auth to the Web UI, changing OTA's auth scheme) — confirm the owner actually wants to revisit that trade-off rather than assuming the existing review missed something.
- Anything needing physical interaction with the device that a chat session can't perform (e.g. touchscreen recalibration).

## 7. Sub-Agent Delegation

The assistant may delegate implementation of a task to another AI coding agent. This is a different trust chain from the assistant writing code directly: the assistant is both director (choosing the model, writing the task) and reviewer of the sub-agent's output, with no independent third party in between.

- **Task selection**: only narrow, well-specified, self-contained tasks tied to a single existing issue. Never delegate anything on the §6 escalation list, or other shared security-critical infrastructure, until the pattern has a longer track record in this project.
- **Model selection**: pick the strongest available model for the task; if quota-limited, define a fallback order in advance rather than deciding mid-task.
- **Isolation**: run the sub-agent in a dedicated, fresh working copy/worktree — never in the assistant's own active workspace.
- **Prompt requirements**: the task prompt must (a) point the sub-agent at this file and require it to follow it, (b) state it is running non-interactively and must make all decisions itself rather than waiting for clarification, (c) explicitly bound the file/directory scope it may touch, (d) require the full local check sequence (`pio test -e native`, `pio check`, `pio run -e esp32-cyd`) before pushing, (e) require it to open the PR itself but explicitly forbid self-approving or merging, (f) forbid AI-attribution commit trailers.
- **Independent verification, not trust**: re-run every check the sub-agent claims passed, inspect the actual diff for scope creep, and assess whether new tests genuinely exercise the claimed behavior — never accept a self-reported "all green" at face value. Confident, articulate self-review claims from a delegated agent are not evidence; only independently reproduced results are.
- **Review visibility**: findings from each verification pass are posted as visible PR comments, not just summarized elsewhere.
- **Fix loop**: if verification finds a real gap, send one scoped follow-up prompt describing exactly what to fix and why, then re-verify. If output is still substantively wrong after a reasonable number of iterations, or the sub-agent has exhausted its own usage limits, the assistant implements the fix/feature itself instead of continuing to iterate.
- **Approval authority**: approving and merging a sub-agent-authored PR requires the owner's explicit authorization, since the assistant directed the author and cannot be the sole independent reviewer. Launching each new autonomous sub-agent session is itself a high-blast-radius action requiring case-by-case confirmation, separate from any standing approve/merge authorization.
- **Shared environment hygiene**: a sub-agent session can leave repo-level state (e.g. a stale committer identity, a leftover branch, an uncommitted experimental file) that silently affects future work without its own override. Check for this after onboarding a new sub-agent tool — this project has already had to clean up orphaned branches and an unrelated committed subproject (`mcp-bitaxe/`) left over from an earlier, undocumented session.
